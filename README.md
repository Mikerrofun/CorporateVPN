# Corporate VPN

Корпоративный VPN-кабинет поверх уже развёрнутого Marzban (Docker, порт 8002, **один** сервер).

## Как это работает

Единая страница входа (`/login`) для всех: email, пароль и код корпорации. По роли,
привязанной к аккаунту, вход сам ведёт в нужный кабинет — ни отдельного адреса для
администратора, ни отдельной формы регистрации нет.

- **Admin** — один платформенный аккаунт. Заходит на `/login` своим email/паролем и
  собственным кодом корпорации, но роль `ADMIN` сразу ведёт его в `/admin`. Там он создаёт
  корпорации и внутри каждой — сотрудников.
- **Создание корпорации** — при нажатии «Создать корпорацию» админ получает: случайный код
  входа (`ACME-7F3K9Q2H`) и **одну** Marzban/Hiddify-подписку, созданную сразу через backend.
  Ссылку и код нужно просто передать сотрудникам.
- **Сотрудник** — заходит на тот же `/login` своим email/паролем + кодом своей корпорации,
  попадает в `/dashboard` и видит **ту же самую** ссылку подписки, что и все остальные
  сотрудники этой корпорации. Персональных VPN-аккаунтов, групп по 5 человек и переключения
  между серверами нет — сервер один, подписка одна на компанию.
- Админ может приостановить (`suspend`) VPN-доступ всей корпорации разом, обновить
  (`rotate`) её ссылку подписки, банить/удалять отдельных сотрудников (это только блокирует
  им вход в кабинет, общая ссылка при этом не меняется).

## Состав

- `web/` — Next.js (App Router) + TypeScript + Tailwind + NextAuth + Prisma/Postgres.
  Владеет всей бизнес-моделью (корпорации, сотрудники, тикеты, аудит-лог) и рендерит обе
  панели: `/dashboard/*` (сотрудник) и `/admin/*` (админ).
- `backend/` — FastAPI (Python). Без бизнес-логики: тонкая обёртка над Marzban API
  (создать общий аккаунт корпорации, ротация ссылки, включить/выключить доступ). Next.js
  вызывает его по внутреннему секрету `INTERNAL_API_SECRET` — из браузера это недоступно.

Поддержка — только на сайте (без Telegram): тикет создаётся в `/dashboard/support`, админ
видит и закрывает его в `/admin/support` (обращения всех корпораций в одном списке).

## Как это стыкуется с существующим Marzban

Marzban уже поднят в Docker на порте 8002 — это единственный VPN-сервер, ничего
разворачивать заново не нужно и управлять серверами/нодами через этот UI нельзя (сознательно
убрано). `.env` по умолчанию указывает `MARZBAN_BASE_URL=http://host.docker.internal:8002`
(работает, если backend в своём docker-compose, а Marzban — в соседнем на том же хосте).

## Первый запуск

```bash
cp .env.example .env
# заполните: POSTGRES_PASSWORD, NEXTAUTH_SECRET (openssl rand -base64 32),
# INTERNAL_API_SECRET (openssl rand -hex 32), MARZBAN_USERNAME/PASSWORD

docker compose up -d --build
docker compose exec web npx prisma migrate deploy
```

Локальная (не-Docker) разработка фронтенда:

```bash
cd web
cp ../.env .env.local        # Next.js сам ищет .env* только внутри web/
npm install
npx prisma migrate dev       # применяет миграции из prisma/migrations
npm run dev
```

## Администратор

Саморегистрации нет и seed-скрипта тоже: admin не хранится в БД, а задаётся в `.env` —
`ADMIN_LOGIN` + `ADMIN_PASSWORD_HASH` (bcrypt-хэш, например
`node -e "console.log(require('bcryptjs').hashSync('пароль', 12))"`). Вход — `/admin/login`.

## VPN: индивидуальный аккаунт на сотрудника

- Каждый сотрудник получает **свой** Marzban-аккаунт и персональный `subscriptionUrl`
  при регистрации. Backend вызывается до записи в БД: если Marzban отказал — регистрация
  отклоняется (`VPN_PROVISIONING_FAILED` / `VPN_BACKEND_UNAVAILABLE`), в БД ничего
  не появляется. Аккаунты в Marzban бесплатны и не лимитированы.
- Группа — логическое объединение (лимиты, инвайты, массовые действия), VPN-полей на ней нет.
- Бан сотрудника — **настоящий**: сначала `setStatus(disabled)` в Marzban (VPN отключается
  мгновенно, даже если ссылка уже в клиенте), затем `User.status = BANNED` — вход и активные
  сессии блокируются (redirect на `/suspended`).
- Групповые действия (suspend/resume/rotate/delete) применяются ко всем участникам группы.
- Перенос между группами не трогает VPN — аккаунт переезжает вместе с сотрудником.
- Дев-режим без Marzban: `VPN_PROVISION_MODE=mock` в `.env`.

Подробности — в `INVITE_CODES_SYSTEM.md`, секция «VPN Provisioning».

## Известные упрощения / что доделать дальше

- Next.js закреплён на `14.2.35` (последний патч ветки 14) и `next-auth@4` — обе библиотеки
  всё ещё тянут несколько moderate/high advisories транзитивно. Полное закрытие потребует
  миграции на Next 15/16 (params становится Promise) или next-auth v5 — сознательно отложено.
- Аналитика на `/admin` дергает `GET /provisioning/usage/<username>` по каждой корпорации
  синхронно при рендере страницы — нормально при небольшом числе корпораций, но при росте
  стоит закэшировать (Redis/периодический снапшот) вместо live-запроса на каждый заход в
  панель.
